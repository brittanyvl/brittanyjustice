CMS BCBS Oncology J-Code Payor Rate Pipeline

Machine-Readable File Processing for Pricing Intelligence

Back to Case Studies

Turning CMS payor machine-readable files into analyzable pricing intelligence for the top oncology J-codes.

Overview

CMS requires payors to publish Machine Readable Files (MRFs) containing negotiated rates. In practice, these files are massive, fragmented across index manifests and in-network rate files, and difficult to use for any targeted analysis.

In this project, I built a pipeline to collect and normalize negotiated rates for the top 50 oncology J-codes from a BCBS payor's CMS files, enabling analysis of how payment rates vary by billing code and provider reference across published plans.

Context / Why It Matters

Oncology drugs represent some of the highest-cost items in healthcare reimbursement. Small changes in negotiated rates can have outsized financial impact for providers, health systems, and risk-bearing entities. While MRFs theoretically improve transparency, the raw format is not decision-ready: it requires engineering work to extract, filter, and structure the rates before any pricing analysis is possible.

The Problem

CMS MRF data is technically public, but operationally difficult:

  • Discovery friction: payors publish an index JSON that references many rate files per plan and market.
  • Scale: in-network files are extremely large JSON documents that can't be loaded into memory safely.
  • Nested structure: negotiated rates live in deeply nested arrays, and provider identities are separated into reference blocks.
  • Join complexity: rate entries often use provider_reference keys requiring separate provider-reference lookups.
  • Analysis readiness: without flattening, even simple questions like "what's the negotiated rate for JXXXX?" are hard to answer.

Solution

I built a two-stage extraction pipeline that turns the payor's CMS artifacts into normalized tables suitable for analysis:

Stage 1: Index Manifest → File Inventory

The index file contains a reporting_structure array mapping plans to in-network file locations. I flatten that structure into a tabular inventory so files can be selected, downloaded, and processed systematically.

  • Extract reporting entity metadata
  • Enumerate plans (name, market type, id type/id)
  • Enumerate in-network file descriptors and URLs
  • Write a clean index table for downstream processing

Stage 2: Streaming In-Network Rate Files → Flat Rate Rows

In-network files are too large for typical JSON loading. I used streaming JSON parsing to safely process them in a memory-efficient way. The extraction focuses on the billing codes of interest (top oncology J-codes), and flattens the nested negotiated rates into row-level records.

  • Stream provider_references to build a provider reference → NPI lookup
  • Stream in_network entries to extract billing code and negotiated rates
  • Output normalized rows: provider reference, billing code, negotiated rate
  • Designed for joinability to provider NPIs and plan metadata

Engineering Highlights

  • Memory-safe processing: used ijson to stream and parse massive JSON files without loading them fully.
  • Separation of concerns: manifest flattening (what files exist) is decoupled from rate extraction (what's inside each file).
  • Join-ready outputs: structured outputs support downstream joins across entity → plan → provider → billing code.
  • Extensible design: the same pipeline pattern can scale to more payors, more billing code sets, and richer negotiated rate attributes.

Decisions Enabled

  • Pricing intelligence: quantify negotiated rate variance for high-cost oncology drugs.
  • Benchmarking: compare rate distributions across billing codes and provider references.
  • Outlier detection: flag unusual rates that may signal contracting anomalies or data quality issues.
  • Data product foundation: create a repeatable extraction pattern for future payors or code sets.

Who This Was For

This project is relevant to healthcare analytics and finance stakeholders who need reimbursement transparency, including:

  • Provider revenue cycle and contracting teams
  • Health system finance and strategic sourcing leaders
  • Analysts investigating negotiated rate variation for high-cost drugs
  • Data engineering teams building transparency pipelines from CMS MRF files

Technical Notes

  • Languages: Python
  • Parsing: ijson for streaming JSON
  • Dataframes: pandas for tabular outputs
  • Output artifacts: flattened index inventory CSV and normalized rate rows
  • Key fields: reporting entity metadata, plan identifiers, file URL, provider reference, NPI lookup, billing code, negotiated rate

The pipeline emphasizes reliability and scale: index discovery is repeatable, and rate extraction is memory-safe for very large MRFs.
